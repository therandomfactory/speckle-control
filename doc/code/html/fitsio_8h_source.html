444 /*=============================================================================
445 *
446 * The following wtbarr typedef is used in the fits_read_wcstab() routine,
447 * which is intended for use with the WCSLIB library written by Mark
448 * Calabretta, http://www.atnf.csiro.au/~mcalabre/index.html
449 *
450 * In order to maintain WCSLIB and CFITSIO as independent libraries it
451 * was not permissible for any CFITSIO library code to include WCSLIB
452 * header files, or vice versa. However, the CFITSIO function
453 * fits_read_wcstab() accepts an array of structs defined by wcs.h within
454 * WCSLIB. The problem then was to define this struct within fitsio.h
455 * without including wcs.h, especially noting that wcs.h will often (but
456 * not always) be included together with fitsio.h in an applications
457 * program that uses fits_read_wcstab().
458 *
459 * Of the various possibilities, the solution adopted was for WCSLIB to
460 * define "struct wtbarr" while fitsio.h defines "typedef wtbarr", a
461 * untagged struct with identical members. This allows both wcs.h and
462 * fitsio.h to define a wtbarr data type without conflict by virtue of
463 * the fact that structure tags and typedef names share different
464 * namespaces in C. Therefore, declarations within WCSLIB look like
465 *
466 * struct wtbarr *w;
467 *
468 * while within CFITSIO they are simply
469 *
470 * wtbarr *w;
471 *
472 * but as suggested by the commonality of the names, these are really the
473 * same aggregate data type. However, in passing a (struct wtbarr *) to
474 * fits_read_wcstab() a cast to (wtbarr *) is formally required.
475 *===========================================================================*/
